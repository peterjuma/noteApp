## 1. Workspace-Scoped Archive (E-014)

**Problem:** The Archive was stored in a single global IndexedDB database (`notesdb_archive`), showing all archived notes from every workspace in one combined list. Users with multiple workspaces expected archive to be scoped like the rest of the app.

**What changed:**

### `src/services/notesDB.js`
- `getArchivedNotes(workspaceFilter)` ‚Äî accepts optional workspace DB name; filters by `sourceWorkspace` field when provided
- `purgeArchive(workspaceFilter)` ‚Äî purges only the specified workspace's archives, or all if omitted
- `getArchiveCount(workspaceFilter)` ‚Äî count scoped to workspace

### `src/App.js`
- `toggleArchiveView` ‚Äî passes `this.state.activeDb` to `getArchivedNotes()` so archive defaults to current workspace
- `onLoadArchive(showAll)` ‚Äî accepts boolean; passes `undefined` (all) or `activeDb` (current workspace) to DB
- `onPurgeArchive(purgeAll)` ‚Äî same scoping pattern
- `handleRestoreNote` / `handlePermanentDelete` ‚Äî update state directly (remove from array) instead of re-fetching, avoiding scope mismatch

### `src/SettingsPanel.js`
- Added `showAllArchive` state toggle
- Archive tab header shows "All workspaces" checkbox; defaults to current workspace only
- Workspace badge appears when filtered to a single workspace
- "Empty Archive" button adapts its confirmation message and behavior to the scope
- Hint text updates: "Showing archive for the current workspace only."
- `useEffect` dependency on `showAllArchive` triggers re-fetch when toggled

## 3. Google Drive Backup (E-016)

**Problem:** Users wanted a cloud backup option beyond GitHub Gist sync. The ask was to explore whether the app could leverage an existing Google sign-in session to back up to Google Drive without requiring a complex integration.

**Analysis:** Google does not allow silent access to Drive data from an existing browser session ‚Äî explicit OAuth consent is always required. However, Google Identity Services (GIS) Token Model provides the closest experience: if the user is already signed into Google in the browser, the OAuth consent popup pre-fills their account and they just click "Allow" ‚Äî no credentials to re-enter.

### `src/services/googleDrive.js` (new file)
Complete Google Drive backup service:
- `loadGisScript()` ‚Äî dynamically loads the GIS client library
- `authorize()` ‚Äî initializes token client with `drive.appdata` scope, triggers consent popup
- `disconnect()` ‚Äî revokes access token
- `driveRequest(url, options)` ‚Äî authenticated fetch wrapper with 401 handling
- `findBackupFile()` ‚Äî searches appDataFolder for existing backup
- `backupToDrive(notes, workspaceName)` ‚Äî serializes notes to JSON, creates or updates `notecache-backup.json` in appDataFolder via multipart upload
- `restoreFromDrive()` ‚Äî downloads and parses the backup file
- `getBackupInfo()` ‚Äî retrieves backup metadata without downloading content
- `setClientId(id)` / `isConfigured()` / `isAuthorized()` ‚Äî configuration management

### `src/SettingsPanel.js`
Added "Google Drive Backup" section to the Sync tab:
- `gdriveClientId`, `gdriveStatus`, `gdriveMessage`, `gdriveAuthorized`, `gdriveBackupInfo` state variables
- OAuth Client ID input field (one-time configuration)
- "Sign in with Google" / "Disconnect" buttons
- "Backup to Drive" / "Restore from Drive" action buttons
- Last backup timestamp display
- Status messages for busy/success/error feedback

### `src/App.js`
- Added `onGdriveRestore` prop handler ‚Äî imports restored notes into current workspace via `db.updateNote()`, refreshes state

## 5. Workspace-Scoped Data Import/Export (E-017)

**Problem:** The Settings > Data tab had limited export capabilities ‚Äî only notes, note metadata, templates, and tags. Missing stores: pinned notes, version history, images, settings. All operations were scoped to the current workspace only with no way to export/import across all workspaces. Additionally, `purgeWorkspace()` was not clearing the `versions` store.

**What changed:**

### `src/services/notesDB.js`
- `CREDENTIAL_KEYS` ‚Äî Set of sensitive settings keys excluded from export/import: `gist_token`, `gist_id`, `sync_enabled`, `sync_interval`
- `exportWorkspaceData(dbName)` ‚Äî Exports all 7 stores as a JSON-serializable object. Images are converted from Blob to base64 using `arrayBuffer()` + `btoa()`.
- `importWorkspaceData(data, dbName)` ‚Äî Merges full workspace data using `put` semantics (upsert). Converts images from base64 back to Blob via `atob()`. Credential keys filtered on import. Returns counts per store.
- **Bug fix**: `purgeWorkspace()` now includes `versions` in the transaction store list ‚Äî previously version history was left behind when purging a workspace.

### `src/App.js`
- `handleNotesBackup(exportAll)` ‚Äî Completely rewritten. Accepts boolean parameter:
  - `false` (default): Exports current workspace ‚Äî individual `.md` files + `_workspace.json` (full data dump of all 7 stores)
  - `true`: Exports all workspaces ‚Äî one folder per workspace, each containing `.md` files + `_workspace.json`, plus a root `_manifest.json` with workspace list and metadata
  - Filename pattern: `notecache_{profile}_{scope}_{date}.zip`
  - Summary alert lists counts for notes, templates, tags, versions, images, pins

- `handleFullBackupImport(event)` ‚Äî New handler with format auto-detection:
  - `_manifest.json` present ‚Üí All-workspaces backup: iterates workspace folders, auto-creates missing workspaces via `db.addWorkspace()`, calls `importWorkspaceData()` per workspace
  - `_workspace.json` present (no manifest) ‚Üí Single-workspace backup: imports directly into current workspace
  - Neither present ‚Üí Legacy format: delegates to existing `handleZipImport()` for backward compatibility
  - Refreshes `allnotes`, `pinnedNotes`, `workspaces` state after import

- Updated `onBackup` prop: changed from `{this.handleNotesBackup}` to `{(exportAll) => this.handleNotesBackup(exportAll)}`
- Added `onFullBackupImport={this.handleFullBackupImport}` prop

### `src/SettingsPanel.js`
- Added `onFullBackupImport` to destructured props
- Added `fullBackupInputRef` (useRef) for the restore backup file input
- Added `dataExportAll` state toggle (boolean)
- Redesigned Data & Archive tab with:
  - **"All workspaces" checkbox** ‚Äî same pattern as Archive tab; toggles `dataExportAll`
  - **Workspace badge** ‚Äî shown when scoped to current workspace
  - **Hint text** ‚Äî dynamically describes scope: "across all N workspaces" or "for the current workspace"
  - **5 action cards** (was 4):
    1. **Full Backup** ‚Äî calls `onBackup(dataExportAll)`, label adapts to scope
    2. **Restore Backup** ‚Äî triggers `fullBackupInputRef`, imports full backup ZIP
    3. **Upload Note** ‚Äî single `.md` import (unchanged)
    4. **Import Notes ZIP** ‚Äî legacy ZIP import (renamed from "Import Archive" to clarify)
    5. **Restore from Gist** ‚Äî unchanged
  - Hidden file input: `<input ref={fullBackupInputRef} accept=".zip" onChange={onFullBackupImport} />`
  - **Danger Zone** ‚Äî updated descriptions to accurately list all data stores being purged: "notes, pins, images, templates, tags, version history, and settings"

## 6. Import Notes from Gist (E-018)

**Problem:** No way to import markdown notes from an arbitrary GitHub Gist. The existing "Restore from Gist" only works with the linked sync gist, not with any gist URL.

### `src/services/gistSync.js`
- `fetchGistNotes(gistInput, dbName)` ‚Äî Fetches any gist (public without auth, private using configured token), extracts `.md`/`.txt`/`.markdown` files, returns `{ notes: [{title, body}], owner, description }`. Handles truncated files by fetching raw URLs.
- `extractGistId(input)` ‚Äî Parses gist URLs (`gist.github.com/user/id`, `gist.githubusercontent.com/...`) and bare hex IDs

### `src/App.js`
- `handleImportFromGist(gistInput)` ‚Äî Calls `fetchGistNotes`, deduplicates via content hash, creates notes, returns `{ success }` for UI feedback
- Passed as `onImportFromGist` prop to SettingsPanel

### `src/SettingsPanel.js`
- Added `onImportFromGist` to destructured props, `Github` to lucide imports
- `gistImportUrl` and `gistImportBusy` state for the input field
- Data tab restructured into **Export** and **Import** subsections with `<h4>` headings
- "Import from Gist" section with URL/ID input, Enter-to-submit, spinning icon during import
- "Restore from Gist" card renamed to "Restore from Sync" for clarity
